# Rafef Tech Development Plan

Date: `2026-06-10`

## Purpose

Move the project from page-by-page fixes to a coordinated implementation plan that:

- unifies Russian and Arabic localization across the whole UI;
- makes inventory availability a single trustworthy source of truth;
- links sales, repair, projects, finance, and events in a consistent business flow;
- reduces future accounting and operational drift.

## Reference Alignment

This plan is based on:

- `skill.zip` -> `rafef-tech-planner/SKILL.md`
- `CURRENT_STATUS.md`
- `docs/project-analysis-bilingual.md`
- the current frontend and backend source tree

The plan keeps the approved decisions from the project brief:

- modular monolith backend;
- shared MySQL database;
- Russian-first UI with Arabic support;
- append-only historical business records;
- one shared inventory used by sales, repair, and projects.

### Architecture findings

1. UI implementation is mixed.
   - Some areas already use `useTranslation('app')` and shared presentation helpers.
   - Many internal pages still use hard-coded Russian strings and local inline page patterns.

2. Inventory truth is fragmented at the consumption layer.
   - Repair and projects already create real stock reservations.
   - Sales approval deducts stock through inventory adjustments.
   - Availability is not yet enforced uniformly across all product-picking interfaces.

3. Finance is not fully linked to sales lifecycle.
   - Sales approval changes stock.
   - Finance transactions and customer ledger entries are still largely manual instead of being generated by sales workflows.

4. Runtime drift exists between source and build artifacts.
   - `backend/src` and `backend/dist` are not guaranteed to match.
   - If the app is started from `backend/dist`, users may see behavior different from the edited TypeScript source.

5. Documentation status is partly stale.
   - Older docs still describe earlier project phases and do not fully reflect the current codebase.

## Root Problem Groups

### Group A: Language and UI consistency

- Hard-coded Russian text still exists in many pages and forms.
- Arabic support is only complete in selected sections.
- Dark theme behavior is inconsistent because styling rules are duplicated across pages.
- Some pages still use older local visual patterns and do not inherit the newer shared behavior.

### Group B: Inventory and operational consistency

- Product availability is shown in multiple places but not guarded uniformly.
- Out-of-stock products are not blocked consistently.
- Reserved stock is logically present in backend flows but not surfaced clearly across all related pages.
- Product selection pages still mix catalog behavior with stock behavior.

### Group C: Accounting and document consistency

- Quotes and invoices are visually closer now, but business effects are not fully centralized.
- Approved invoices should produce linked financial effects, not only stock effects.
- Returns, refunds, stock restoration, and customer ledger need one clear lifecycle map.

### Group D: Shared shell and page contracts

- Not all pages use the same toolbar, status, empty, pagination, and error patterns.
- Search, sorting, date formatting, and list counts are still implemented ad hoc in multiple places.
- This makes localization and theme fixes slower than they need to be.

## Execution Strategy

Do not continue page-by-page in isolation.

Instead, work in cross-cutting batches where each batch includes:

- translation;
- theme/dark-mode cleanup;
- business flow wiring;
- shared UI cleanup;
- verification for both Russian and Arabic.

## Phase Plan

### Phase 1: Stabilize runtime and source of truth

Goals:

- remove source/dist ambiguity;
- finish inventory truth propagation;
- define one business status map per module.

Tasks:

1. Normalize backend runtime:
   - choose one supported development path: `backend npm run dev`;
   - rebuild `backend/dist` whenever release/start mode is used;
   - verify that frontend points to the intended backend instance.

2. Inventory source-of-truth hardening:
   - compute `available = on_hand - active_reservations`;
   - expose the same availability meaning from catalog and inventory endpoints;
   - prevent adding zero-availability products in POS, repair parts, and project materials;
   - cap requested quantity by actual available stock for invoice flows.

3. Shared business lifecycle map:
   - `quote` -> no stock, no finance, no ledger;
   - `draft invoice` -> no stock, no finance, no ledger;
   - `approved invoice` -> stock deduction + finance transaction + optional customer ledger;
   - `return` -> stock restore + linked refund/credit flow;
   - `repair/project reservation` -> reserved only until consume/release step.

### Phase 2: Build shared UI and localization infrastructure

Goals:

- stop duplicating page chrome and text patterns;
- make RU/AR support predictable across all modules.

Tasks:

1. Introduce shared page contracts:
   - common page header structure;
   - common toolbar layout;
   - common list summary text;
   - common empty/error/loading blocks;
   - common local date and money formatting helpers.

2. Translation normalization:
   - define module namespaces for `customers`, `catalog`, `inventory`, `repair`, `sales`, `finance`, `projects`, `creative`, `settings`, `reports`, `events`;
   - move hard-coded page strings into locale files;
   - keep Russian default labels and full Arabic equivalents;
   - keep sorting helpers aware of Arabic, Latin, and Cyrillic order.

3. Theme normalization:
   - move repeated dark-mode surface rules to shared tokens and global CSS;
   - remove one-off bright panels and legacy light fills from internal pages;
   - verify `rtl` layout behavior where spacing or alignment depends on direction.

### Phase 3: Finish the commerce backbone

Priority modules:

- `catalog`
- `inventory`
- `sales`
- `finance`

Tasks:

1. Catalog and stock:
   - expose stock-aware product cards and tables;
   - show on-hand, reserved, available, and threshold consistently;
   - split "catalog browsing" from "stock picking" where needed.

2. POS and sales:
   - finalize stock guardrails;
   - keep quote behavior non-destructive;
   - connect approval to financial posting;
   - connect returns to stock reversal and refund/credit logic;
   - keep print logic separate from invoice creation.

3. Finance:
   - generate finance transactions from approved invoices;
   - generate ledger entries only when customer debt/credit is involved;
   - keep walk-in cash sales out of customer ledger;
   - keep work session and daily closing totals tied to real transactions.

### Phase 4: Finish service operations backbone

Priority modules:

- `repair`
- `projects`
- `creative`
- `crm`

Tasks:

1. Repair:
   - translate all screens;
   - show reserved parts clearly;
   - define when reserved parts are consumed vs released;
   - connect repair status changes to event visibility.

2. Projects:
   - show reserved materials, installed assets, and project status clearly;
   - translate pages and shared labels;
   - keep project material reservations visible in inventory.

3. Creative:
   - unify language and theme;
   - decide which stock-linked materials need inventory hooks and which stay service-only.

4. CRM:
   - finish localization;
   - keep language-aware customer sorting centralized and reused;
   - connect customer financial summary and activity history to real module outputs.

### Phase 5: Finish admin, reporting, and event visibility

Priority modules:

- `reports`
- `events`
- `settings`
- `integrations`

Tasks:

1. Reports:
   - ensure reports exclude quotes from real sales;
   - ensure stock reports reflect reservations and approved movements only;
   - ensure financial reports derive from finance records, not UI assumptions.

2. Events:
   - emit events from sales approvals, returns, stock reservations, releases, repair readiness, and project milestones;
   - keep ticker messages localized.

3. Settings and admin:
   - finish translation coverage;
   - remove demo-only wording from user-facing settings when inappropriate;
   - keep language/theme controls authoritative for the whole UI.

## Immediate Work Order

1. Finish runtime normalization and rebuild discipline.
2. Finish inventory availability propagation across POS and stock-picking flows.
3. Introduce shared localized list/page patterns.
4. Convert `customers`, `catalog`, and `inventory` to the shared pattern.
5. Link approved sales to finance transactions and customer ledger logic.
6. Convert `repair` and `projects` with the same stock-aware rules.
7. Sweep the remaining modules for translation and dark-theme parity.

## Acceptance Criteria

The project should be considered operationally coherent only when all of the following are true:

1. Every visible page supports Russian and Arabic without hard-coded leftovers.
2. Availability shown in UI matches backend stock reality.
3. Out-of-stock items cannot be added where real stock is required.
4. Repair and project reservations reduce available stock consistently.
5. Approved invoices affect both inventory and finance in a traceable way.
6. Quotes do not affect stock, finance, or ledger.
7. Returns and refunds create correct reverse effects.
8. Reports match approved business records only.
9. The dark theme uses shared surfaces and no random bright legacy panels remain.
10. `npm run build`, `npm run typecheck`, `backend npm run build`, and `backend npm run typecheck` pass after each batch.

## Notes About the Current `available = 0` Symptom

The repository state suggests this issue should be treated as a cross-layer problem, not as a single-field bug:

- the business database already contains real stock quantities;
- source logic and built logic are not fully synchronized;
- some screens still consume product data without complete stock guardrails;
- the UI still needs one unified stock-aware product selection contract.

This is why the fix should be completed as part of Phase 1, before further UI polish.
