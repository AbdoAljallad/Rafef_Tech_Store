# Выпускная квалификационная работа

## Тема

Разработка внутренней веб-системы управления деятельностью магазина «Tech Store» для Rafef Tech Store

## Введение

В последние годы цифровизация стала не просто удобным дополнением к работе малого бизнеса, а необходимым условием его устойчивого развития. Даже небольшой магазин ежедневно обрабатывает десятки взаимосвязанных операций: ведет учет товаров, принимает оплату, оформляет продажи, работает с клиентами, контролирует складские остатки, фиксирует расходы и формирует отчеты. Когда все эти процессы распределены между таблицами, бумажными записями и разрозненными программами, руководитель получает не систему управления, а набор фрагментов, между которыми постоянно приходится вручную восстанавливать связи.

Для технического магазина такая ситуация особенно сложна. В отличие от обычной торговой точки, здесь работа не ограничивается продажей товара. В одном бизнесе одновременно сосуществуют розничная торговля, ремонт устройств, прием заказов на дизайн и печать, выполнение монтажных проектов, учет материалов, взаимодействие с клиентами и учет финансовых операций. Из-за этого типовая кассовая программа покрывает лишь небольшой участок реальной деятельности, а остальная информация продолжает жить отдельно.

В качестве предметной области в данной работе рассматривается деятельность магазина Rafef Tech Store, расположенного в Каире, Египет. Магазин работает с 2017 года и совмещает несколько направлений: продажу технических товаров, ремонт оборудования, услуги дизайна и печати, а также выполнение проектов по установке систем видеонаблюдения и компьютерных сетей. Такой формат делает предприятие хорошим примером для разработки прикладной информационной системы, ориентированной не на абстрактную модель магазина, а на реальные рабочие процессы.

До начала разработки значительная часть учета велась с помощью электронных таблиц, бумажных заметок и отдельных программных инструментов. Это позволяло фиксировать отдельные операции, но не давало целостной картины бизнеса. Становилось трудно быстро определить фактические остатки, отследить движение товара, связать продажу с клиентом и финансовой операцией, сопоставить расход материалов с конкретным проектом или ремонтом. Подготовка отчетности занимала лишнее время, а качество управленческих решений зависело от того, насколько аккуратно владелец вручную собирал данные из разных источников.

Актуальность темы обусловлена необходимостью создания единой внутренней веб-системы, способной объединить ключевые процессы магазина в одной цифровой среде. Такая система должна не просто хранить данные, а поддерживать логику бизнеса: связывать продажи со складом, ремонт с резервированием запчастей, проекты с материалами, а финансовые операции с первичными документами. Кроме того, современный веб-подход позволяет обеспечить удобный доступ к системе из локальной рабочей среды без усложнения повседневной эксплуатации.

Целью выпускной квалификационной работы является разработка внутренней веб-системы управления деятельностью магазина Rafef Tech Store, обеспечивающей централизованный учет данных, поддержку основных бизнес-процессов и возможность дальнейшего расширения функциональности.

Для достижения поставленной цели необходимо решить следующие задачи:

1. Проанализировать предметную область и специфику работы магазина.
2. Исследовать существующий способ учета и выявить его основные ограничения.
3. Сформулировать функциональные и нефункциональные требования к системе.
4. Спроектировать архитектуру веб-системы, включающую клиентскую часть, серверную часть и базу данных.
5. Разработать структуру данных для хранения информации о товарах, клиентах, продажах, ремонтах, проектах и финансовых операциях.
6. Реализовать основные программные модули системы.
7. Обеспечить разграничение прав доступа и основу для дальнейшего масштабирования.
8. Подготовить систему к интеграции с внешними сервисами автоматизации.
9. Выполнить тестирование основных пользовательских сценариев и оценить практические результаты разработки.

Объектом исследования является деятельность технического магазина Rafef Tech Store.

Предметом исследования являются методы, программные средства и архитектурные решения, применяемые при разработке внутренней веб-системы управления магазином.

Практическая значимость работы состоит в том, что создаваемая система ориентирована на реальные задачи действующего предприятия. Ее внедрение позволяет сократить объем ручного учета, повысить прозрачность операций, упростить контроль за складом и ускорить получение управленческой информации.

## 1. Аналитическая часть

### 1.1. Анализ предметной области

Rafef Tech Store представляет собой небольшой технический магазин сервисного типа. Его деятельность строится вокруг нескольких взаимосвязанных направлений, каждое из которых влияет на другие. Продажа товара уменьшает складские остатки и должна отражаться в истории клиента. Ремонт использует запчасти и услуги, а значит, требует учета материалов, стоимости работ и статусов выполнения. Проекты по установке оборудования задействуют товары со склада, монтажные площадки и отдельный контроль исполнения. Финансовый блок, в свою очередь, должен связывать кассовые и безналичные операции с продажами, возвратами, расходами и рабочими сменами.

Иными словами, магазин функционирует как компактная, но достаточно насыщенная информационная среда, где бизнес-процессы нельзя качественно автоматизировать по отдельности. Если хранить сведения о товарах, клиентах, ремонте и финансах независимо друг от друга, возникают разрывы, которые быстро превращаются в управленческие риски.

### 1.2. Характеристика деятельности магазина Rafef Tech Store

Особенность рассматриваемого предприятия заключается в том, что значительная часть процессов долгое время держалась на личном опыте владельца и его практических знаниях. Такой подход типичен для малого бизнеса: пока объем операций невелик, многие связи между заказами, клиентами и остатками удерживаются в памяти или в неформальных заметках. Однако по мере роста количества товаров, ремонтов и услуг подобная модель начинает терять устойчивость.

В случае Rafef Tech Store учет должен охватывать не только стандартные торговые операции, но и работу с устройствами клиентов, запчастями, расходом материалов в проектах, созданием печатных документов, контролем поступления денежных средств и администрированием сотрудников. Это означает, что системе требуется не один справочник товаров и не одна кассовая форма, а целая совокупность модулей, объединенных общей бизнес-логикой.

С практической точки зрения для магазина особенно важны следующие особенности:

1. Один и тот же товар может одновременно участвовать в продаже, ремонте и проекте.
2. Складской остаток должен различать фактическое наличие, резерв и доступное количество.
3. Финансовые операции должны быть связаны с конкретными документами и бизнес-событиями.
4. Пользователи системы работают с разными ролями и не должны иметь одинаковый уровень доступа.
5. Интерфейс должен быть достаточно простым, чтобы им можно было пользоваться в повседневной работе без отдельной длительной подготовки.

### 1.3. Анализ существующего способа учета и выявление проблем

До разработки веб-системы магазин использовал смешанный способ учета. Часть данных фиксировалась в электронных таблицах, часть хранилась в бумажных записях, часть обслуживалась отдельным программным обеспечением. Такой подход не является редкостью, но именно в нем скрывается источник большинства организационных затруднений.

Проблема заключалась не только в том, что данные находились в разных местах. Гораздо серьезнее было отсутствие единой логики работы. Если, например, товар использовался в ремонте, это не всегда сразу отражалось в общей картине складских остатков. Если оформлялась продажа, то последующая сверка с движением денег и историей клиента требовала дополнительных действий. Если выполнялся монтажный проект, расход материалов мог фиксироваться отдельно от общей товарной базы.

## 2. Проектная часть

### 2.1. Общая архитектура веб-системы

Проект реализован как full-stack монорепозиторий, в котором клиентская часть, серверная часть, миграции базы данных, инфраструктурные файлы и документация развиваются совместно. Такой подход удобен для дипломного проекта и малого бизнеса, поскольку позволяет хранить всю систему в одном рабочем контуре и при этом не терять логическое разделение на уровни.

Серверная часть построена по модели модульного монолита. Это означает, что приложение разворачивается как единый backend-сервис, но внутри состоит из независимых доменных модулей: `auth`, `crm`, `catalog`, `inventory`, `repair`, `sales`, `finance`, `creative`, `projects`, `reports`, `integrations`, `admin` и других. Выбранный подход удачно сочетает архитектурную аккуратность и практичность: модульная структура уже есть, а операционная сложность микросервисов для данного масштаба пока не нужна.

### 2.2. Проектирование клиентской части

Клиентская часть расположена в каталоге `src/` и организована по доменному принципу. В папке `routes` находятся маршрутные страницы, в `modules` сосредоточена прикладная логика по направлениям бизнеса, а в `shared` вынесены общие UI-компоненты, API-клиент, тема, права доступа и локализация.

Такое проектирование позволяет избежать хаотичного смешения кода. Экран продаж относится к маршрутам, типы и API-запросы продаж — к модулю `sales`, а общие таблицы, модальные окна и навигация переиспользуются из общего слоя. В результате интерфейс не превращается в набор отдельных страниц, написанных по разным правилам.

Важной особенностью клиентской части является ролевая навигация. Пользователь видит и открывает только те разделы, которые соответствуют его полномочиям. Еще одна значимая черта — задел под многоязычность. Основной рабочий язык интерфейса на текущем этапе — русский, однако архитектура уже использует `i18next` и допускает переключение языковых ресурсов.

### 2.3. Проектирование серверной части

### 4.3. Проверка корректности работы базы данных

Корректность базы данных оценивалась через выполнение миграций, создание связанных сущностей и проверку изменений после бизнес-операций. Особое внимание уделялось сценариям, в которых данные должны синхронно изменяться сразу в нескольких контурах: например, при продаже товара, резервировании материалов или возврате документа.

С точки зрения дипломного проекта особенно важен тот факт, что база данных не выступает пассивным хранилищем. Она является основой связанного учета, где изменения в одной подсистеме отражаются на других. Именно это позволяет говорить о разработанной системе как о полноценном инструменте управления, а не как о наборе независимых форм.

### Приложение Г. Инструкция по запуску проекта

Разработанная система может быть запущена двумя основными способами: через локальную инфраструктуру Docker Compose и через раздельный запуск frontend- и backend-частей в режиме разработки. Первый вариант является наиболее удобным для первичного развертывания проекта, поскольку автоматически поднимает все основные сервисы. Второй вариант подходит для доработки и отладки отдельных частей системы.

#### Г.1. Минимальные требования к окружению

Для запуска проекта на рабочем компьютере необходимы следующие компоненты:

- `Docker Desktop` или другой совместимый Docker-движок;
- `Node.js` и `npm`;
- терминал `PowerShell`;
- доступ к локальным портам `80`, `3000`, `3306`, `5173` и `8080`.

Если система запускается только через Docker Compose, локальная установка Node.js не является обязательной, однако для раздельного запуска frontend и backend она необходима.

#### Г.2. Подготовка файлов окружения

Перед первым запуском необходимо подготовить переменные окружения для базы данных и прикладных сервисов.

Для MySQL создается файл:

```powershell
Copy-Item infrastructure/mysql/.env.example infrastructure/mysql/.env
```

Типовое содержимое этого файла:

```env
MYSQL_ROOT_PASSWORD=rafef_root_password
MYSQL_DATABASE=rafef_tech
MYSQL_USER=rafef_user
MYSQL_PASSWORD=rafef_password
```

Для frontend при необходимости можно создать локальный конфигурационный файл:

```powershell
Copy-Item .env.example .env.local
```

При локальной работе frontend с backend в режиме разработки используется следующая конфигурация:

```env
VITE_API_BASE_URL=http://localhost:3000
VITE_USE_MOCK_AUTH=false
```

Backend использует файл:

```text
backend/.env
```

Если файл отсутствует, его можно создать на основе шаблона:

```powershell
Copy-Item backend/.env.example backend/.env
```

Для локальной базы данных используются значения:

```env
NODE_ENV=development
PORT=3000
APP_BASE_URL=http://localhost:5173
CORS_ORIGIN=http://localhost:5173
MYSQL_HOST=localhost
MYSQL_PORT=3306
MYSQL_USER=rafef_user
MYSQL_PASSWORD=rafef_password
MYSQL_DATABASE=rafef_tech
SESSION_COOKIE_NAME=rt_session
SESSION_TTL_HOURS=12
SESSION_SECRET=change-this-local-secret
SEED_ADMIN_USERNAME=admin
SEED_ADMIN_PASSWORD=admin123
```

#### Г.3. Запуск проекта через Docker Compose

Наиболее удобный способ развертывания системы заключается в использовании `Docker Compose`. В этом случае из корня проекта необходимо выполнить команду:

```powershell
docker compose up -d
```

После запуска будут подняты следующие сервисы:

- `MySQL` на порту `3306`;
- `backend` на порту `3000`;
- `frontend` на порту `5173`;
- `Caddy` на порту `80`;
- `Adminer` на порту `8080`.

Особенностью текущей конфигурации является то, что контейнер backend при старте автоматически выполняет команду:

```bash
npm run db:setup
```

Эта команда, в свою очередь, последовательно запускает миграции и начальное заполнение базы данных:

```bash
npm run migrate
npm run seed
```

Таким образом, при первом запуске Docker Compose структура базы данных создается автоматически, а в системе появляется учетная запись администратора по умолчанию.

#### Г.4. Раздельный запуск backend и frontend

Если требуется отдельная разработка frontend и backend без полного Docker-окружения, проект можно запускать по частям.

Сначала необходимо поднять локальную базу данных:

```powershell
docker compose up -d mysql adminer
```

После этого запускается backend:

```powershell
cd backend
npm install
npm run db:setup
npm run dev
```

Затем в корне проекта запускается frontend:

```powershell
cd ..
npm install
npm run dev
```

В этом режиме backend обычно доступен по адресу:

```text
http://localhost:3000
```

А frontend:

```text
http://localhost:5173
```

#### Г.5. Проверка корректности запуска

После старта системы рекомендуется убедиться, что контейнеры и веб-сервисы работают корректно.

Проверка состояния Docker-сервисов:

```powershell
docker compose ps
```

Проверка backend:

```powershell
Invoke-RestMethod http://localhost:3000/api/health
```

Ожидаемый ответ должен содержать признаки успешной работы сервера и базы данных, например:

```json
{"status":"ok","database":"ok"}
```

При использовании Caddy открыть систему можно по адресу:

```text
http://localhost
```

При прямом запуске frontend:

```text
http://localhost:5173
```

Для визуальной проверки структуры базы данных можно использовать `Adminer`:

```text
http://localhost:8080
```

Параметры подключения к базе данных в `Adminer`:

```text
System: MySQL
Server: mysql
Username: rafef_user
Password: rafef_password
Database: rafef_tech
```

После успешного запуска пользователь может войти в систему под учетной записью администратора, созданной при начальном заполнении базы данных:

```text

Пароль: admin123
```

Эти данные предназначены только для начального локального использования. При реальном внедрении системы пароль администратора должен быть изменен.

#### Г.7. Дополнительные служебные команды

Для сопровождения проекта в процессе разработки могут использоваться следующие команды.

Просмотр журналов Docker-сервисов:

```powershell
docker compose logs -f backend
docker compose logs -f frontend
docker compose logs -f caddy
```

Перезапуск отдельных сервисов:

```powershell
docker compose restart backend
docker compose restart frontend
docker compose restart caddy
```

Остановка всей инфраструктуры:

```powershell
docker compose down
```

Повторный запуск:

```powershell
docker compose up -d
```

Если по какой-либо причине в базе данных отсутствуют необходимые таблицы или нарушена инициализация структуры, можно повторно выполнить:

```powershell
docker compose exec backend npm run db:setup
```

Это делает проект удобным как для демонстрации в рамках выпускной квалификационной работы, так и для дальнейшего сопровождения, тестирования и поэтапного расширения функциональности.
